# 网络拓扑与配置说明

本文面向不熟悉 Linux 网络虚拟化的读者，介绍本部署方案中的网络架构、相关术语和配置原理。

---

## 现实类比

想象一个办公室局域网：

- 办公室里有一台**交换机**，所有电脑的网线都插在上面
- 每台电脑配置一个**内网 IP**（如 192.168.1.x），互相可以 ping 通
- 交换机本身不关心 IP，它只认 **MAC 地址**（网卡硬件地址），负责在不同端口之间转发数据帧

我们的 QEMU 虚拟化环境就是在宿主机里**模拟出这个办公室**。

---

## 核心角色

```
                         宿主机 (WSL2)
                    ┌──────────────────────┐
                    │                      │
                    │   br0 网桥(交换机)     │
                    │   172.16.0.1          │
                    │   ┌──┬──┬──┐          │
                    │   │  │  │  │          │
                    │  tap0 tap1 tap2       │
                    │   │  │  │  │          │
                    │  ┌┴──┴──┴─┐│          │
                    │  │  QEMU  ││          │
                    │  │ VM1    ││          │
                    │  │.101    ││          │
                    │  └────────┘│          │
                    └────────────┼──────────┘
                                 │
                          172.16.0.0/24 网段
```

### 1. br0 网桥 —— 虚拟交换机

Linux 里 `bridge` 就是一个**软件实现的交换机**。我们用 `brctl` 创建它：

```bash
ip link add br0 type bridge
ip addr add 172.16.0.1/24 dev br0
```

它的作用和物理交换机完全一样：插在它上面的设备可以通过 MAC 地址直接互相通信。因为给它配了 IP，宿主机自己也接入这个局域网。

### 2. tap0 / tap1 / tap2 —— 虚拟网线

`tap` 是 Linux 内核提供的一种**虚拟以太网设备**。可以把它理解为：

> 一头连着 VM 的虚拟网卡，另一头插到交换机 (br0) 上。

创建方式：

```bash
ip tuntap add dev tap0 mode tap
ip link set tap0 master br0     # 插入交换机
```

每台 VM 对应一根"网线"：

| 设备 | 连接关系 |
|------|---------|
| tap0 | VM1 虚拟网卡 ↔ br0 |
| tap1 | VM2 虚拟网卡 ↔ br0 |
| tap2 | VM3 虚拟网卡 ↔ br0 |

QEMU 启动时通过 `-netdev tap,ifname=tap0` 把这根网线交给 VM1 使用：
```
VM1 虚拟网卡 ←→ tap0 ←→ br0 ←→ tap1 ←→ VM2 虚拟网卡
```

### 3. "二层互通" —— 在同一台交换机下面

网络分层（简化版）：

```
┌──────────────────────────┐
│ 应用层    HTTP / TiKV RPC │  ← 端口号 (2379, 20160...)
├──────────────────────────┤
│ 传输层    TCP / UDP      │
├──────────────────────────┤
│ 网络层    IP 地址         │  ← "三层"，靠 IP 路由
│           172.16.0.x     │
├──────────────────────────┤
│ 数据链路层 MAC 地址 / ARP  │  ← "二层"，靠交换机转发
│           52:54:00:...   │
├──────────────────────────┤
│ 物理层    网卡 / tap      │
└──────────────────────────┘
```

**二层互通**的含义：所有设备在同一个交换机下面，不需要经过 IP 路由，数据帧通过 MAC 地址直接送达。就像同一台交换机上的三台电脑，插上网线就能通信，不需要设置网关。

VM1 (172.16.0.101) ping VM2 (172.16.0.102) 的过程：

1. VM1 发出 ARP 广播："谁的 IP 是 172.16.0.102？"
2. 广播经过 tap0 → br0 → tap1 到达 VM2
3. VM2 回复自己的 MAC 地址 `52:54:00:12:34:02`
4. VM1 用这个 MAC 地址封装数据帧发给 VM2
5. 交换机 (br0) 看到目标 MAC，从 tap1 端口转发出去

整个过程只涉及二层，不需要路由器或网关。

## 完整网络拓扑

```
┌─────────────────────────────────────────────────────────────────┐
│                        宿主机 WSL2                               │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                   br0 (172.16.0.1)                       │    │
│  │                   虚拟交换机                              │    │
│  │            tap0         tap1         tap2               │    │
│  └──────────────┬────────────┬────────────┬─────────────────┘    │
│                 │            │            │                      │
│  ┌──────────────┴──┐  ┌──────┴──────┐  ┌──┴──────────────┐     │
│  │    VM1          │  │    VM2      │  │    VM3          │     │
│  │  tikv-vm1       │  │  tikv-vm2   │  │  tikv-vm3       │     │
│  │                 │  │             │  │                 │     │
│  │ 网卡1 (tap):    │  │ 网卡1(tap): │  │ 网卡1(tap):     │     │
│  │ 172.16.0.101   │  │ 172.16.0.102│  │ 172.16.0.103   │     │
│  │ MAC:52:54:..01 │  │ MAC:..02    │  │ MAC:..03       │     │
│  │                 │  │             │  │                 │     │
│  │ 网卡2 (user):   │  │ 网卡2(user):│  │ 网卡2(user):    │     │
│  │ DHCP 自动获取   │  │ DHCP 自动   │  │ DHCP 自动       │     │
│  │                 │  │             │  │                 │     │
│  │ PD1 + TiKV1     │  │ PD2 + TiKV2 │  │ PD3 + TiKV3     │     │
│  │ :2379 :2380     │  │ :2379 :2380 │  │ :2379 :2380     │     │
│  │ :20160 :20180   │  │ :20160 :20180│  │ :20160 :20180  │     │
│  └─────────────────┘  └─────────────┘  └─────────────────┘     │
│                                                                 │
│  宿主机访问入口:                                                   │
│  localhost:2379 ─→ VM1:2379  (PD API - 端口转发)                 │
│  172.16.0.101   ─→ VM1:22    (SSH - 桥接直连)                   │
│  172.16.0.102   ─→ VM2:22    (SSH - 桥接直连)                   │
│  172.16.0.103   ─→ VM3:22    (SSH - 桥接直连)                   │
└─────────────────────────────────────────────────────────────────┘
```

### 每台 VM 有两张网卡

| 网卡 | 类型 | 用途 | 连接方式 |
|------|------|------|---------|
| **enp0s2** (tap) | 内网 | PD 集群内部通信、TiKV 节点间 Raft 复制、SSH | 通过 tap 接入 br0 网桥，二层互通 |
| **enp0s3** (user) | 外网 | 仅 PD API 端口转发、尝试访问互联网 | QEMU 用户态网络栈，NAT + 端口转发 |

### 内网通信（走 ens3 / br0）

| 通信路径 | 端口 | 说明 |
|---------|------|------|
| PD1 ↔ PD2、PD3 | 2380 | Raft 心跳、Leader 选举、成员变更 |
| TiKV → PD 集群 | 2379 | 心跳注册、Region 调度、TSO 时间戳请求 |
| TiKV1 ↔ TiKV2 ↔ TiKV3 | 20160 | Raft 日志复制、Snapshot 同步 |

### 宿主机访问

| 命令 | 说明 |
|------|------|
| `curl localhost:2379/pd/api/v1/health` | 查询 PD 健康状态（端口转发） |
| `ssh ubuntu@172.16.0.101` | SSH 到 VM1（桥接直连） |
| `ssh ubuntu@172.16.0.102` | SSH 到 VM2（桥接直连） |
| `ssh ubuntu@172.16.0.103` | SSH 到 VM3（桥接直连） |

> **注意**：SSH 走桥接网络直连而非 QEMU 端口转发。QEMU SLIRP 用户态网络栈存在 SSH 横幅握手不兼容问题，TCP 连接能建立但 SSH 协议层无法正常通信。桥接网络是标准的二层以太网，不存在此问题。

### VM 访问外网（代理中继）

VM 默认没有外网访问能力，因为宿主机的代理（通常跑在 Windows 侧 `127.0.0.1:7897`）无法被 VM 直接访问。

`setup-network.sh` 会在宿主机桥接 IP 上启动一个 **socat 中继**：

```
VM (172.16.0.101)
  → export http_proxy=http://172.16.0.1:8889
    → 172.16.0.1:8889 (socat relay)
      → 127.0.0.1:7897 (宿主机/Windows 代理)
        → 外网
```

cloud-init 将代理配置写入 `/etc/profile.d/tikv-proxy.sh`，登录后自动生效。`no_proxy` 排除内网地址避免 PD/TiKV 通信走代理。

| 访问路径 | 网络 | 说明 |
|---------|------|------|
| VM ↔ VM (PD/TiKV) | br0 桥接 | 内网直连，不走代理 |
| VM → 宿主机 SSH | br0 桥接 | 172.16.0.x:22 |
| VM → 外网 | br0 → socat → 代理 | 172.16.0.1:8889，中继到宿主机代理 |
| 宿主机 → VM PD API | localhost 端口转发 | localhost:2379 → VM1:2379 |

## 数据流示例

以 TiKV 客户端（宿主机上的程序）写一条数据为例：

```
客户端 (宿主机)
  │
  │ 1. 连接 PD API: localhost:2379
  │    └─ QEMU port forward → VM1 ens4 → VM1 PD 进程
  │       返回：数据应写入 TiKV Region Leader 所在节点
  │
  │ 2. 连接 TiKV: localhost:20160  
  │    └─ QEMU port forward → VM1 ens4 → VM1 TiKV 进程
  │       Raft Leader 接收写入
  ▼
TiKV1 (VM1) ──Raft 复制──→ TiKV2 (VM2) via ens3 → br0 → tap1
            ──Raft 复制──→ TiKV3 (VM3) via ens3 → br0 → tap2

三条 Raft 复制走 br0 内网，不占用端口转发带宽。
```

## 术语速查

| 术语 | 一句话解释 | 在本项目中的作用 |
|------|-----------|----------------|
| **bridge (网桥)** | 软件模拟的交换机 | br0 让 3 台 VM 像插在同一个交换机上一样互通 |
| **tap** | 虚拟以太网设备 | VM 的虚拟网线，一头接 VM 网卡，一头插到 br0 |
| **二层 (L2)** | MAC 地址层，同一广播域 | tap+br0 使 VM 间直接通过 MAC 地址通信，无需路由 |
| **三层 (L3)** | IP 地址层，需路由转发 | 如果 VM 在不同网段，就需要路由器转发 IP 包 |
| **MAC 地址** | 网卡硬件地址，`52:54:00:12:34:01` | QEMU 虚拟网卡的唯一标识，交换机靠它转发帧 |
| **ARP** | IP → MAC 的解析协议 | VM1 想知道 172.16.0.102 对应的 MAC 地址时发出广播 |
| **NAT / 端口转发** | 网络地址转换 | QEMU user 模式把 localhost 端口映射到 VM 内部端口 |
| **QEMU user 模式** | QEMU 内置的轻量网络栈 | 无需 root/内核模块，提供 NAT 上网和端口转发 |
| **DHCP** | 自动分配 IP 地址 | user 模式的 enp0s3 网卡自动获取 IP（不一定能上网） |
| **静态 IP** | 手动配置固定 IP | enp0s2 网卡通过 bootcmd 配置静态 IP（172.16.0.101-103），确保地址不变 |
| **QEMU SLIRP** | QEMU 用户态网络实现 | 实现 NAT + 端口转发，但 SSH 协议层不兼容，仅用于 PD API 端口转发 |
| **bootcmd** | cloud-init 最早执行的阶段 | 在 systemd 启动前直接用 `ip addr add` 配网，mask 掉会卡住的 wait-online 服务 |
| **Raft** | 分布式一致性协议 | PD 集群和 TiKV Region 均使用 Raft 保证数据一致 |
| **TSO** | Timestamp Oracle | PD 为分布式事务分配全局唯一递增时间戳 |

## 相关文件

| 文件 | 作用 |
|------|------|
| `setup-network.sh` | 创建 br0 网桥和 tap0/tap1/tap2 设备 |
| `cloud-init/vm*-user-data` | 为每台 VM 配置静态 IP 和 /etc/hosts |
| `start-vms.sh` | 以 `-netdev tap` 启动 VM，绑定到对应的 tap 设备 |
| `config/pd*.toml` | PD 的 advertise-peer-urls 指向 172.16.0.x:2380 |
| `config/tikv*.toml` | TiKV 的 pd.endpoints 指向全部 3 个 PD 的 2379 |
